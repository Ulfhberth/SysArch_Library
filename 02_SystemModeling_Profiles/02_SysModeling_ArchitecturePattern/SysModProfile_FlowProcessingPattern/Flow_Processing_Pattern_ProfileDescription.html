Flow Processing Pattern

SysML Profile for Media Flow-Oriented System Architectures

üìã Architecture Pattern Explanation

The Flow Processing Pattern is a specialization of the classic Pipes-and-Filters architectural pattern and describes a structured method for building systems where media (information, energy, or materials) are "processed" in a chain.

üéØ Core Concept

The media to be processed flow sequentially through a series of filters (processing components), each fulfilling a clearly defined task, while interfaces of the filters act as connecting communication pathways.

Fundamental Principles

  • Media Processor: Individual, independent processing units that transform, validate, or enrich media
  • Media Interface: Defined connections that couple processors together and transport media flow
  • Sequential Media Flow: Each processor receives input media from one interface and passes processed output media through another interface
  • Modular Composition: Complex processing chains emerge through chaining or reordering individual, reusable processors
üí° Suitability: The pattern is particularly suitable for flow or streaming systems where modular structure, flexible composition, and loose coupling are essential goals.

‚úÖ Benefits of the Pattern

üîó Loose Coupling & Better Replaceability

Processors only know their inputs and outputs; otherwise they are independent. This makes them easier to replace, recombine, and reuse in other flow chains.

üìà High Extensibility

New functionality emerges by "adding media processors" rather than rebuilding existing processors. Particularly advantageous for additional validation, variant handling, or logging/monitoring.

‚ö° Scaling per Stage

When deploying, "hotspots" (complex sub-flow chains) can be specifically scaled without completely rebuilding the entire flow chain.

üß™ Testability and Maintainability

Small, clearly defined processors are easier to test in isolation and maintain (Separation of Concerns). Each processor has a clear responsibility.

üîÑ Asynchronicity & Temporal Decoupling

Interfaces enable temporal decoupling of producers and consumers and allow different transport modes (synchronous/asynchronous/streaming) ‚Äì particularly helpful in event/stream processing.

‚ö†Ô∏è Risks and Disadvantages

üîå Schema/Media Format Coupling (Invisible Dependencies)

Despite loosely coupled media processors, strong coupling can arise through the format of the respective media at the interfaces.

Remedy: Deploy converters in specific sub-flow chains.

‚è±Ô∏è Latency Overhead from Many Stages

Each component plus interface (queue/serialization) costs processing time.

Caution: Critical analysis necessary for hard real-time requirements and complex flow chains.

üçù Danger of "Flow Chain Spaghetti"

Many branches and side-channels (e.g., additional taps, "Taps") can make the overall architecture confusing.

Remedy: Strict governance and rules:

  • Naming Conventions and Stereotypes
  • Clear interface definition (contracts)
  • Unique versioning (especially in variant handling)

üéØ Typical Application Areas

üì° Real-Time Signal Processing

Audio, video, or sensor signals are processed step by step (e.g., noise suppression ‚Üí feature extraction ‚Üí classification).

üåä Data Stream Systems

Log and event processing in monitoring systems, Big Data pipelines, and ETL processes. Continuous processing of large data volumes.

üîß Software with Multiple Transformation Stages

Compiler designs, image processing, protocol stacks ‚Äì everywhere data must be progressively transformed.

üè≠ IoT and Industry 4.0

Processing sensor data in production environments with various analysis and aggregation stages.

üìä Business Intelligence & Analytics

Data preparation, transformation, and aggregation in multi-stage analysis pipelines.

üì¶ Package Overview

The Flow Processing Pattern SysML Profile provides a formal modeling foundation for applying the Flow Processing Pattern in SysML-based system architectures. It extends SysML with specialized stereotypes that make the characteristic elements of the pattern ‚Äì processors, interfaces, and media flows ‚Äì explicitly modelable.

üí° Goal: The profile enables architects to model media flow-based architectures consistently, comprehensibly, and with tool support. Through clear typing, the architecture becomes more understandable and better analyzable.

¬´Stereotype¬ª FlowChain
extends Block

Description: The FlowChain stereotype describes an abstract, sequential processing chain within the pattern. It represents the highest level of abstraction in which media flow step by step through an ordered sequence of independent processing units (filters).

Application: The FlowChain defines the sequence, coupling, and consistency of the processing processors without specifying their internal logic. It serves as an architectural framework for modeling modular, extensible, and reusable media flow chains.

Tagged Values:

FlowChainKind : String [1]
Description: Classification of the flow chain type. Describes the topological structure of the processing chain.
Values:
‚Ä¢ linear (sequential chain, classic pipeline structure)
‚Ä¢ branching (chain with branches, parallel processing paths)
‚Ä¢ converging (multiple parallel paths merge into common processor)
‚Ä¢ dag (Directed Acyclic Graph, complex structure)
‚Ä¢ circular (contains feedback loops, iterative refinement)
‚Ä¢ hybrid (combination of different structure types)
StreamType : String [1]
Description: Type of data stream. Defines how media flows through the chain.
Values:
‚Ä¢ continuous (unlimited, continuous data stream without defined end)
‚Ä¢ batch (limited data volumes processed in batches)
‚Ä¢ event-driven (processing triggered by specific events)
‚Ä¢ micro-batch (hybrid: continuous stream divided into very small batches)
‚Ä¢ request-driven (processing on explicit request)
ProcessingType : String [1]
Description: Processing mode. Defines the temporal behavior of flow chain execution.
Values:
‚Ä¢ synchronous (sequential processing, blocking and deterministic)
‚Ä¢ asynchronous (non-blocking processing with temporal decoupling)
‚Ä¢ parallel (simultaneous processing of multiple media units)
‚Ä¢ pipelined (classic pipeline model, optimized throughput)
‚Ä¢ hybrid (combination of different processing modes)

¬´Stereotype¬ª FlowProcessor
extends Block

Description: The stereotype describes an independent processing unit within the pattern. It represents a logically self-contained component that receives inputs via defined interfaces, transforms, analyzes, or enriches them, and passes results to subsequent components.

Application: FlowProcessors are independent of each other, interchangeable, and free from assumptions about upstream or downstream processing steps. They form the fundamental building blocks of the media flow chain.

Tagged Values:

Purpose : String [1]
Description: Describes the functional purpose of the component. Makes its role in the flow chain explicit.
Values:
‚Ä¢ validation (verifies media for correctness and rule compliance)
‚Ä¢ transformation (converts media from one format to another)
‚Ä¢ enrichment (extends media with additional information)
‚Ä¢ aggregation (combines multiple media units and calculates aggregated values)
‚Ä¢ filtering (selects relevant media and discards irrelevant ones)
‚Ä¢ normalization (standardizes media to uniform format)
‚Ä¢ deduplication (identifies and removes duplicates)
‚Ä¢ routing (directs media to different processing paths)
‚Ä¢ splitting (divides one medium into multiple separate media)
‚Ä¢ merging (combines multiple media into one)
‚Ä¢ buffering (temporarily buffers media for decoupling)
‚Ä¢ monitoring (monitors media and captures metrics)
‚Ä¢ encryption (encrypts or decrypts media)
‚Ä¢ compression (compresses or decompresses media)
‚Ä¢ orchestration (coordinates complex processing sequences)
Statefulness : Boolean [1]
Description: Indicates whether the component is stateful or stateless.
Values:
‚Ä¢ false (stateless: no state stored between processings, easy scaling)
‚Ä¢ true (stateful: stores state information, required for aggregations)
Note: Stateless components are easier to scale and test. Stateful components require state management and possibly persistence.
ProcessRate : String [1]
Description: Describes processing speed or throughput rate. Important for performance analysis and scaling decisions.
Values:
‚Ä¢ real-time (processing with minimal latency, milliseconds)
‚Ä¢ near-real-time (processing with low delay, seconds to few minutes)
‚Ä¢ batch-oriented (processing in batches at defined times)
‚Ä¢ high-throughput (optimized for maximum throughput, >10,000 msg/sec)
‚Ä¢ medium-throughput (moderate processing rate, 1,000-10,000 msg/sec)
‚Ä¢ low-throughput (low processing rate, <1,000 msg/sec)
‚Ä¢ variable (processing rate fluctuates, requires elastic scaling)
Note: Concrete numerical values (e.g., '1000 msg/sec') can be specified as free text.

¬´Stereotype¬ª FlowInterface
extends Interface

Description: The stereotype describes a standardized interface for providing or receiving media within the pattern. It defines direction, structure, and type of media input and output.

Application: FlowInterfaces encapsulate data communication, promote loose coupling of processors, and enable consistent, interchangeable, and comprehensible modeling of media flows.

Tagged Values:

Ordering : String [1]
Description: Defines the order of media transmission. Important for correct stream processing.
Values:
‚Ä¢ FIFO (First-In-First-Out, fair queue processing, predictability)
‚Ä¢ LIFO (Last-In-First-Out, stack principle, newest information first)
‚Ä¢ priority-based (based on assigned priority, critical vs. normal processing)
‚Ä¢ unordered (no defined order, opportunistic, maximum throughput)
‚Ä¢ timestamp-based (sorted by timestamp, Event-Time Processing)
‚Ä¢ round-robin (uniform rotating distribution, load balancing)
BufferCapacity : Integer [1]
Description: Maximum number of media units that can be buffered in the interface. Determines backpressure behavior and decoupling between producer and consumer.
Typical Ranges:
‚Ä¢ 0 = No buffer (synchronous handover)
‚Ä¢ 1-100 = Small buffer (low decoupling)
‚Ä¢ 100-10,000 = Medium buffer (standard for many scenarios)
‚Ä¢ >10,000 = Large buffer (high decoupling, higher memory requirement)
‚Ä¢ -1 = Unlimited buffer (caution: memory growth)
TransportMode : String [1]
Description: Type of media transmission. Determines temporal communication behavior.
Values:
‚Ä¢ synchronous (blocking communication, sender waits for confirmation)
‚Ä¢ asynchronous (non-blocking, temporal decoupling, higher throughput)
‚Ä¢ streaming (continuous data flow, real-time streams)
‚Ä¢ request-reply (bidirectional, sender expects response)
‚Ä¢ fire-and-forget (unidirectional without confirmation, maximum throughput)
‚Ä¢ publish-subscribe (one-to-many communication, loose coupling)
‚Ä¢ polling (consumer periodically queries, pull-based)
‚Ä¢ callback (event-based, callback on availability)
RateLimit : String [1]
Description: Restriction of transmission rate. Important for resource management and quality-of-service guarantees.
Values:
‚Ä¢ unlimited (no rate restriction, maximum throughput)
‚Ä¢ bandwidth-limited (based on data volume per time, e.g., '10 MB/sec')
‚Ä¢ message-limited (based on message count, e.g., '100 msg/sec')
‚Ä¢ token-bucket (allows bursts, average rate controlled)
‚Ä¢ leaky-bucket (smooths traffic to constant rate)
‚Ä¢ adaptive (dynamic adjustment to system conditions)
‚Ä¢ fixed-window (limit for fixed time windows, simple implementation)
‚Ä¢ sliding-window (sliding time window, smoother traffic)
Note: Concrete numerical values (e.g., '1000 msg/sec', '10 MB/sec') can be specified as free text.

Relationships between Stereotypes

Structural Dependencies

A FlowChain contains multiple FlowProcessors.

FlowProcessors possess ProxyPorts (inputs and outputs, standardized by SysML).

A ProxyPort is typed by a FlowInterface.

ProxyPorts are connected to other ProxyPorts via Connectors.

Typical Pipeline Structure

  1. FlowChain defines the overall structure and processing logic
  2. If needed, FlowChains can be recursively decomposed
  3. FlowProcessors are instantiated in the FlowChain
  4. Each FlowProcessor has ProxyPorts with typed FlowInterfaces (Input/Output)
  5. Connectors link the ProxyPorts

‚úÖ Best Practices

  • Use clear Naming Conventions (e.g., prefix "fi_" for FlowProcessor ProxyPorts with FlowInterfaces)
  • Document media formats at each FlowInterface via FlowProperty
  • Version FlowChain definitions for better traceability
  • Use the Statefulness attribute consciously for scaling decisions
  • Define FlowInterface characteristics explicitly (Ordering, BufferCapacity, etc.)
